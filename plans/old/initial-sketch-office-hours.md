# Design: Pick Me A Dinner — a context tool for the nightly family-dinner decision

Generated by /office-hours on 2026-05-15

## Problem Statement

Deciding what the family eats for dinner is a recurring nightly friction. The hard
part is not cooking or ordering — it is *choosing*, and choosing badly because you
can't remember what you've already eaten this week or which restaurants are even in
play. The result is the frustrated "I don't know, what do you want" loop.

This is a small personal web app that removes that friction. It is explicitly **not**
an oracle and **not** a meal planner. In the user's own words:

> "It is a sharp tool to do one job: help me pick family dinner every night. I am
> still picking, and I need context about what we as a family have eaten recently
> and possible restaurants to do so."

The human stays the decider. The app's only job is to put the right context in front
of them fast: recent-meal memory + restaurant options, ranked so variety and
freshness are obvious. A 15-second decision instead of a 10-minute argument.

## What Makes This Cool

The signature feature is **explainable dinner memory**. Every option in the ranked
list carries one plain-English reason for where it sits: "No fish in 18 days",
"Italian was 2 nights ago — maybe skip", "Family favorite · last had 28 days ago".
The app is not smart at you; it shows its work. That is what makes the ranking
trustworthy and the tool feel sharp rather than gimmicky.

Second: it treats **restaurants and home-cooked meals as first-class equals** in one
catalog and one log. Almost no app in the meal-planning space does this — they are
recipe planners where eating out is a "failure" of the plan. Here, "El Comal" and
"sheet-pan salmon" compete in the same ranked list on the same terms.

## Constraints

- Personal, single-household app. No multi-tenant, no real user accounts.
- Hosted web app, reachable from any device, protected by a single shared password.
- Responsive and genuinely ergonomic on **both** phone and desktop:
  - Phone is the primary surface — dinner is picked in the kitchen around 5pm.
  - Desktop is first-class too — catalog management and ordering-out research
    typically happen on a larger screen.
- Small by intent. ~4 screens, ~4 tables, one interesting algorithm. Resist growth
  into recipes / grocery lists / nutrition / weekly planning.
- Typed end-to-end (TypeScript).

## Premises

1. The nightly family-dinner decision is real, recurring friction. Doing nothing
   means continuing the frustrated "what do you want" loop indefinitely.
2. The fix is **context, not automation**. The human stays the decider; the app
   surfaces recent-meal memory and restaurant options. No oracle, no auto-plan.
3. **Pick = log, one tap.** Choosing a meal logs it as tonight's dinner — no
   separate logging step. Accept that a logged "plan" approximates what was actually
   eaten; input friction is the documented killer of these apps.
4. **Free-form shared tags** drive both behaviors. Similarity = shared tags. The
   suggestion engine ranks the catalog using per-item recency (anti-repeat) and
   per-tag recency (variety enforcing).
5. **Restaurants and home meals are first-class equals** in one unified catalog and
   one log.
6. Personal single-household app, responsive on phone and desktop, protected by a
   single shared password (one env var, no real accounts), deployed as a hosted web
   app reachable from any device.

## Cross-Model Perspective

(it had not seen this conversation). Key points:

  "explainable dinner memory" — a "tonight dashboard" where every ranked option
  explains itself. This matched the design direction and is now a first-class
  requirement, not a nice-to-have.
- **Key insight quoted back.** Codex flagged the same sentence as the heart of the

  decision surface."
- **50% starting point.** Codex suggested Mealie (open-source self-hosted meal

  more than it saves. Read Mealie's data model as a reference; do not fork it.
- **Challenged premise.** Codex challenged the original "no accounts" premise: a
  hosted web app with zero auth exposes family data to anyone with the URL. This

- **Radical scope discipline.** Codex's explicit skip-list (no recipes, grocery
  lists, AI, weekly planning, nutrition, per-person preferences, maps/hours) matches
  the wedge and is folded into Non-Goals below.

Where Claude diverges from Codex: Codex picked SQLite; the user chose Postgres, and
that is the better call here — it removes the persistent-volume constraint and makes
managed hosting (Neon/Supabase/Railway) clean.

## Approaches Considered

### Approach A: One small server, server-rendered
A single tiny backend process serving server-rendered HTML with a database.
Effort S, Risk Low. Fewest dependencies, ships fastest, but live tag autocomplete and
the explanation-chip "tonight dashboard" feel clunkier without a component model, and
extending later is more manual wiring.

### Approach B: Next.js App Router + Drizzle + Postgres  (CHOSEN)
Next.js App Router, TypeScript, Drizzle ORM, Postgres, Tailwind, responsive component
UI, server actions for pick=log. Effort M, Risk Low. Typed data model end-to-end;
the component model serves the explanation-chip dashboard and dual phone/desktop
ergonomics directly.

### Approach C: PocketBase backend + static PWA
PocketBase (one binary = DB + REST API + auth + admin UI) as the entire backend, with
a static PWA frontend. Effort S-M, Risk Med. Backend is nearly free, but the ranking —
the actual hard part — gets zero help from PocketBase, so the boilerplate saved is
boilerplate there was little of.

## Recommended Approach

**Approach B, Postgres-backed.** Final stack:

- **Next.js (App Router) + TypeScript** — typed end-to-end; server actions make
  pick=log a one-liner; component UI keeps the dual phone/desktop layout clean.
- **Postgres** via **Drizzle ORM** — typed queries and migrations. Postgres (vs
  SQLite) removes the persistent-volume constraint and deploys cleanly on managed
  hosts.
- **Tailwind CSS** — responsive utility styling.
- **Auth** — single shared password from an env var, checked once per device, stored
  in a cookie. No accounts, no user table.

### Data model

```
items
  id            uuid pk
  name          text
  kind          enum('home','restaurant')
  url           text null          -- menu / delivery link, mainly for restaurants
  notes         text null
  active        boolean default true   -- inactive = hidden from Tonight + Catalog
                                       -- default list, still shown in Log history
  created_at    timestamptz

tags
  id            uuid pk
  name          text               -- app trims + lowercases every tag on input
                                    -- UNIQUE INDEX on lower(name) enforces
                                    -- case-insensitive uniqueness — no citext
                                    -- extension needed, no host-gating risk

item_tags
  item_id       uuid fk -> items  ON DELETE CASCADE
  tag_id        uuid fk -> tags   ON DELETE CASCADE
  pk (item_id, tag_id)

dinner_log
  id            uuid pk
  item_id       uuid fk -> items  ON DELETE RESTRICT   -- see deletion rule below
  eaten_on      date               -- multiple items per date allowed (see pick=log)
  note          text null
  created_at    timestamptz
  UNIQUE (item_id, eaten_on)        -- the SAME item can't be logged twice on one
                                    -- date (double-tap guard); DIFFERENT items on
                                    -- one date is fine — that's a real dinner
```

`dinner_log` is the single source of truth for all recency. Per-item recency =
most recent `eaten_on` for that item. Per-tag recency = most recent `eaten_on`
across all items carrying that tag (computed via `item_tags` join).

**Lifecycle rules:**

- **Deactivate, don't delete logged items.** An item that has any `dinner_log`
  row cannot be hard-deleted (`ON DELETE RESTRICT` enforces this). The Catalog
  screen offers "Archive" → sets `active = false`. Archived items vanish from
  Tonight and the default Catalog list but still render correctly in Log history.
  An item with zero log rows may be hard-deleted.
- **`pick = log` appends; multiple items per date are allowed.** A real dinner is
  sometimes two restaurants, or takeout plus some home cooking. Tapping "Pick
  tonight" inserts a `dinner_log` row for today; picking another item the same
  evening adds a second row — the date legitimately carries both. The only thing
  blocked is logging the *same* item twice on one date: `pick = log` upserts on
  `(item_id, eaten_on)`, so an accidental double-tap of one item is a harmless
  no-op. Mis-picked an item you didn't actually eat? Remove it on the Log screen.
  The Log screen groups entries by date; a date may show one or more.
- **Tags are merged, never deleted.** The Tags screen exposes rename and merge
  only — no standalone delete. A typo tag is fixed by renaming or merging it into
  the right one. (`ON DELETE CASCADE` on `item_tags.tag_id` exists only to keep a
  merge's final `DELETE tag B` clean.) The whole merge runs in a **single
  transaction** so a mid-merge failure cannot leave partial state.
- **Tag edits are not retroactive.** Per-item and per-tag recency are computed
  from an item's *current* tags, not the tags it had when a past dinner was
  logged. Removing "fish" from salmon means past salmon dinners stop feeding
  "fish" recency. This is intentional and simplest; if it ever feels wrong,
  revisit — but not in v1.

### Screens

1. **Tonight** (home, primary) — ranked catalog. Each row: name, quiet Home/Restaurant
   badge, one explanation chip (the plain-English reason), tag chips each showing
   per-tag recency ("fish 18d", overdue tags in accent color), and a one-tap
   "Pick tonight" action that logs a `dinner_log` row for today (multiple picks
   per date allowed — see the `pick = log` rule). Sticky filter zone:
   All/Home/Restaurant segment + tri-state tag filter chips.
   **Empty state:** if the catalog has zero items, Tonight shows a short prompt
   ("Add your first meals →") linking to the Catalog screen, instead of an empty
   list.
2. **Log** — reverse-chronological dinner history, grouped by date; a date may
   carry more than one entry (multi-restaurant or restaurant + home nights).
   Edit/delete any logged entry.
3. **Catalog** — add/edit home meals and restaurants; attach tags inline. The
   desktop-heavy screen.
4. **Tags** — lightweight: autocomplete + rename/merge. No taxonomy management UI.

### Access gate

Before any screen renders, the app checks for a valid auth cookie. If absent, it
shows a single password field. On submit, the entered value is compared
(constant-time) to the `APP_PASSWORD` env var. On match, an `httpOnly` cookie is
set holding a **signed token** — an HMAC over a fixed payload using a second env
var `APP_SECRET` as the key, never a guessable constant — so the cookie cannot be
forged without the secret. The token itself is not time-bound; the long expiry
(e.g. 180 days) is enforced by the cookie's `Max-Age` attribute. Wrong password →
inline error, no lockout (personal app). No logout UI in v1; clearing the cookie
is sufficient.

### Tag filtering (tri-state)

The Tonight filter bar uses tappable tag chips, each with three states:
**off → include → exclude → off**. Include = show only items carrying that tag.
Exclude = hide items carrying that tag. Include and exclude filters combine
(e.g. show "quick", hide "spicy"). The kind segment (All / Home / Restaurant) and
the tag filters **AND together** — an item must satisfy the kind filter *and* all
include tags *and* none of the exclude tags. A hint line states the active filter
in words.

### Ranking algorithm (v1)

The Tonight list is the full active catalog (after filters), sorted descending by a
score combining two signals: **anti-repeat** (per-item recency) and **variety
enforcer** (per-tag recency). Higher score = more "due", floats to the top.

**Where it runs.** Computed in TypeScript, not SQL. For a personal catalog (tens of
items, hundreds of log rows) the whole dataset is tiny. The Tonight server component
fetches three things — `items` (active), `item_tags`, and `dinner_log` — and computes
scores in a pure function. This keeps the logic unit-testable and the explanation-chip
derivation trivial. No multi-CTE SQL needed.

**Core helper, with explicit null handling:**

```
CAP = 60                       // days; also the substitute for "never"

daysSince(date | null):
    if date is null:  return CAP            // never eaten / never used
    return min( CAP, todayInDays - date )   // capped so "never" can't dominate forever
```

`eaten_on` is a SQL `date` (no timezone). Before subtraction, both `eaten_on` and
"today" are converted to integer **epoch-days in the app's local timezone**
(`APP_TZ` env var, e.g. `America/Los_Angeles`) so "today" is the household's
calendar day, not the server's UTC day. `lastEaten(item)` and `lastTagUse(t)` are
the per-item / per-tag recency dates defined in the Data model section above; both
return `null` when the item / tag has never appeared in `dinner_log`.

**Score:**

```
itemScore(item):
    anti_repeat = daysSince(lastEaten(item))           // 0..CAP

    tagDays     = [ daysSince(lastTagUse(t)) for t in tags(item) ]
    variety     = tagDays.length === 0
                    ? anti_repeat                      // tagless item: variety
                                                       // term mirrors anti-repeat
                                                       // (no empty-avg / NaN)
                    : mean(tagDays)                    // 0..CAP

    return  W_ITEM * anti_repeat
          + W_TAG  * variety
          + W_FAV  * favoriteBonus(item)
```

Both `anti_repeat` and `variety` are in the same unit (capped days, 0..CAP), so they
combine cleanly.

**Starting weights** (live in one `ranking.config.ts`, tuned by feel after real use):

```
W_ITEM = 1.0      // anti-repeat
W_TAG  = 1.0      // variety enforcer
W_FAV  = 0.0      // favoriteBonus OFF in v1 — see below
```

**`favoriteBonus`** is **off in v1** (`W_FAV = 0`). When enabled later it must be
expressed in the same capped-day unit as the other terms — define it as a normalized
day-equivalent (e.g. `min(CAP, totalTimesEaten * k)`), never a raw count, so the
weighted sum stays unit-consistent. Defer until there is real log data to tune against.

**Explanation chip** — one per row, derived deterministically from the score:

- If `W_TAG * variety >= W_ITEM * anti_repeat` (tag term dominates, ties included —
  variety is the headline feature): the chip names the **single tag with the
  largest `daysSince`** → "No fish in 18 days".
- Otherwise (item term dominates): the chip names the item's own recency →
  "Last had 28 days ago".
- The "Family favorite · ..." chip variant is **deferred with `favoriteBonus`** —
  both the bonus and its favorite-qualifying threshold are out of v1 (`W_FAV = 0`).

The chip is a product requirement: if the ranking can't explain itself in one
plain-English line, it isn't done.

**"Overdue" tag styling.** A tag chip renders in the accent color when
`daysSince(lastTagUse(t)) >= OVERDUE_THRESHOLD`, where `OVERDUE_THRESHOLD = 14`
(also in `ranking.config.ts`). Purely visual; does not affect the score.

**Cold start.** On a brand-new install with zero `dinner_log` rows, every item
scores `(W_ITEM + W_TAG) * CAP` — a flat tie. Tonight then falls back to ordering by
`items.name` (stable, predictable) until enough log history exists to differentiate.

## Open Questions

These are genuine feel-calls to settle after real use — not unresolved spec gaps.
(Edge cases the reviewer flagged — null recency, tagless items, cold start, tag-merge
collisions — are now fully specified in the algorithm and schema sections above.)

- **Tag recency on a brand-new item.** A new home meal tagged "pasta" inherits
  "pasta last used 9 days ago" from the log even though the dish itself is new. The
  algorithm treats this as correct (you *did* have pasta recently) — confirm it
  feels right in practice after a couple weeks, or raise `W_ITEM` relative to
  `W_TAG` for never-eaten items.
- **Skipped days.** Days with no dinner logged (away, leftovers, forgot to open
  the app) simply leave a gap in `dinner_log` — no special handling needed; the
  recency math treats a gap as just more days elapsed. (Multiple items per date
  is now a settled rule, not an open question — see the `pick = log` lifecycle
  rule above.)
- **Tag-hygiene aggressiveness.** The `lower(name)` unique index + the rename/merge
  action prevent case-drift and let you clean up. How often you actually need to
  merge is a feel call after a few weeks of free-form tagging.

## Success Criteria

- Opening "Tonight" on a phone and picking a dinner takes under 15 seconds.
- Every ranked row shows a correct, plain-English explanation chip.
- The same log and catalog are visible and editable from both phone and desktop.
- After ~2 weeks of real use, the variety enforcer visibly surfaces neglected
  categories (the user notices "huh, we really haven't had X").
- The user actually opens it most evenings — the true test of whether friction is
  low enough.

## Non-Goals (explicit — protect the wedge)

No recipes. No ingredient lists or grocery lists. No nutrition or calorie tracking.
No AI / LLM suggestion. No forward weekly meal planning. No per-family-member
preference modeling. No maps / opening-hours / live-availability integrations. No
real multi-user accounts. If a feature pulls toward "meal planner," it is out.

## Distribution Plan

- **Deliverable:** a hosted Next.js web app.
- **Hosting (v1 decision): Railway.** The Next.js app and a Railway-managed
  Postgres instance live in one project. Railway is chosen over Vercel/Fly
  specifically because it gives a first-class **release/start command** — the
  clean place to run migrations (see CI/CD). No persistent volume needed; that
  was the point of choosing Postgres over SQLite. (Vercel + Neon remains a valid
  fallback, but is not the v1 plan.)
- **Access:** `APP_PASSWORD` + `APP_SECRET` + `APP_TZ` env vars; signed cookie
  remembered per device.
- **CI/CD:** push-to-deploy from the Git repo (Railway does this natively).
  **Migrations:** Railway runs `drizzle-kit migrate` as the release/pre-start
  command on every deploy, before the app process starts. One mechanism, decided —
  no manual step, no build-script workaround. No separate release pipeline needed.
- **Backup:** rely on the managed Postgres provider's automated backups; optionally
  a periodic `pg_dump` for peace of mind.

## The Assignment

Before writing a single line of code, spend 20 minutes seeding your real catalog by
hand — on paper or in a notes file. List every home-cooked meal you actually rotate
through (aim for 15-25) and every restaurant in play (aim for 8-12). Tag each one as
you go, using whatever free-form words feel natural.

Two reasons this is the assignment and not "go build it":

1. **It de-risks the cold-start problem.** This app is worthless with an empty
   catalog — the ranking has nothing to rank. The seed list is the fuel, and you
   want it ready the moment the Catalog screen exists.
2. **It stress-tests premise 4 before you commit code to it.** Free-form tagging
   only works if your tags stay roughly consistent. Twenty minutes of real tagging
   will tell you immediately whether you naturally converge ("mexican" every time)
   or drift ("mexican", "mexican-food", "tex-mex") — and that tells you how much
   tag-hygiene tooling the Tags screen actually needs.

If the tagging feels messy after 20 minutes, that's a finding — bring it back.
